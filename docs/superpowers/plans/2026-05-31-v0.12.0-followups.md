## Follow-up 1 â€” Fix the pre-existing Alembic stamp bug (not schema-flip related)

**Symptom:** the integration suite errors in fixture setup, all tests:

```
INSERT INTO alembic_version (version_num) VALUES ('2026_05_05_0010_per_tenant_shadow_copy')
â†’ asyncpg StringDataRightTruncationError: value too long for type character varying(32)
```

**Root cause (pre-existing, unrelated to v0.12.0):**
- `mypalace/database.py` `_ensure_alembic_stamp()` hardcodes
  `alembic_version.version_num VARCHAR(32)`, but the project's revision
  ids exceed 32 chars (`2026_05_05_0010_per_tenant_shadow_copy` = 38,
  `..0011_emotional_contexts` = 34, `..0013_backfill_tenant_id` = 34).
- `LATEST_ALEMBIC_REVISION` in `mypalace/database.py:69` is **stale** â€”
  pinned at `2026_05_05_0010_per_tenant_shadow_copy` while the migration
  head is now `2026_05_31_0013_backfill_tenant_id`. On a fresh DB,
  `init_db()` create_all's every table then stamps `0010`; a subsequent
  `alembic upgrade head` would then try to re-run `0011`/`0012`'s
  `op.create_table` against tables that already exist.

**Why CI didn't catch it:** `.github/workflows/release.yml` runs only
`pytest tests/` (unit) and `pytest mypalace_client/tests/`. The
integration suite (`-m integration`) is never run in CI.

**Proposed fix:**
1. Widen the stamp column: in `_ensure_alembic_stamp`, create
   `version_num VARCHAR(64)` (and, if a real `alembic upgrade` ever
   creates the table, ensure `alembic/env.py` /
   `context.configure(version_table_...)` agrees â€” confirm what width
   real Alembic uses for this project, since 0.11.x shipped 34/38-char
   ids).
2. Update `LATEST_ALEMBIC_REVISION` to the actual head
   (`2026_05_31_0013_backfill_tenant_id`) so fresh-deploy stamping +
   `alembic upgrade head` are a no-op, not a conflict.
3. Add a unit/lint guard that `LATEST_ALEMBIC_REVISION` equals the single
   Alembic head (catches future staleness).

## Follow-up 2 â€” Rework integration fixtures for mandatory schema mode

The integration suite was written for table mode and needs adapting now
that schema isolation is always on. Do this with live containers (Docker
available) and iterate.

**Findings (from `tests/integration/conftest.py`):**
- The suite runs **auth-disabled** (`PALACE_AUTH_DISABLED=true`,
  `PALACE_DEFAULT_TENANT_ID=test` from `tests/conftest.py`). There is **no
  `admin_headers` / key-minting fixture** (the original plan's T8 wrongly
  assumed one). Tests target tenants via direct service calls or the
  `tenant_id` query param.
- With auth disabled, `AuthMiddleware` seats `current_tenant="test"` for
  every HTTP request â†’ HTTP-path writes now land in the `test.*` schema.
  But direct service calls in test bodies set no contextvar â†’ they hit
  `public.*` (which Approach A keeps + filters, so they still work).
- `_truncate_tables` (autouse) deletes from `public.*` with no tenant
  scope â†’ it will **not** clean `test.*` data written via HTTP paths,
  risking cross-test leakage.
- `_ensure_default_tenant()` now (v0.12.0, T3) replicates the `test`
  schema automatically, so the test schema exists â€” no extra provisioning
  needed in the fixture.

**Proposed work:**
1. Make the test data location consistent. Simplest: wrap the
   `_truncate_tables` deletes in `tenant_scope("test")` so cleanup hits
   `test.*` (public tables `ApiKey`/`AuditLog`/`Tenant` still resolve via
   `public` on the search_path), AND ensure test bodies that call
   services directly either run under `tenant_scope("test")` or are fine
   on `public.*`. Decide one canonical location and make both paths agree.
2. Add the isolation + cross-tenant-aggregation tests originally drafted
   for T8, adapted to the auth-disabled model (create tenants via
   `POST /v1/admin/tenants`; drive per-tenant writes/reads via the
   `tenant_id` param / cross-tenant admin behavior):
   - tenant A's memory is invisible to tenant B (search_path isolation),
   - `stats?tenant_id=ALL` reports each tenant's own counts,
   - export/import round-trips into the right schema.
3. Requires Follow-up 1 fixed first (otherwise fixture setup errors before
   any test runs).
