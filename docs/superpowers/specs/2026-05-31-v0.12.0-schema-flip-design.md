# v0.12.0 — finish phase 12 (schema-mode flip) — design

**Status:** approved (Joshua, 2026-05-31). Ready for implementation plan.
**Author:** Claude.
**Scope:** two repos — `mypalace` (server + client) and `mypalclara`
(consumer). The bulk of the work is server-side.

This spec finishes phase 12 (per-tenant Postgres schemas). Phases 12.1–
12.3a already shipped in v0.11.0 (runtime plumbing, tenant lifecycle,
Alembic 0010 shadow-copy); the default mode is still `table` and the
`PALACE_TENANT_SCHEMA_MODE` flag still exists. This spec is **12.3b**:
make schema isolation mandatory and remove the flag.

See `docs/per-tenant-schemas-design.md` for the original phase-12 design.
This spec **deviates** from that doc's §3 12.3b in one deliberate way:
it does **not** drop the `tenant_id` columns or the duplicate `public.*`
rows in v0.12.0 (see Approach A below).

---

## Locked decisions

| Decision | Choice |
|---|---|
| Cutover style | **Maintenance window** (brief planned downtime). Online rolling cutover stays out of scope (design doc §7). |
| mypalclara change | **Minimum** — bump the `mypalace-client` pin only; no code changes. |
| Data safety | **Production-grade** — full export backup + row-count verify gate; treat as real Clara data. |
| 12.3b approach | **A: flip + keep columns** — defer the destructive `DROP COLUMN` + filter deletion to a later v0.13.0. |

---

## Guiding principle

`search_path` **is** the isolation boundary. Once schema mode is
mandatory, every per-tenant query resolves bare table names to
`<tenant>.<table>`, so a forgotten `WHERE tenant_id = ...` can no longer
leak across tenants — it just hits the (correct) tenant schema. That
means the `tenant_id` columns and the ~288 explicit `tenant_id` filters
scattered across ~25 files become **harmless redundancy**. We keep them
in v0.12.0 and clean them up later.

Consequence: v0.12.0 delivers the *breaking* change (default flip + flag
removal) while leaving `public.*` data and `tenant_id` columns intact,
which keeps rollback cheap (downgrade + flip back to `table` reads the
untouched `public.*`).

---

## Approach A vs B (why A)

**A — flip + keep columns (chosen).** Make `schema` the default, remove
the flag, backfill `tenant_id` in the shadow-copied per-tenant rows so
the kept filters stay correct, and fix cross-tenant aggregation to loop
schemas. Defer the `DROP COLUMN` + `public.*` purge + filter deletion to
v0.13.0. Smallest, safest, near-reversible diff.

**B — full hard cut (not chosen).** Additionally drop `tenant_id`
columns + `public.*` rows and delete/rewrite all ~288 filters now.
Matches design doc §3; gets the index-size win and a clean codebase; by
far the largest and riskiest single release, with the column drop as the
true point of no return.

Rationale for A: the security guarantee comes from `search_path`, not
from the column drop. The drop only buys clarity + index size, which is
irrelevant at Clara's scale and not worth putting the irreversible,
high-churn step inside the release we're flipping.

## Part 1 — Palace server

### 1.1 Default flip + flag removal
- `mypalace/config.py`: change `tenant_schema_mode` default `table` →
  `schema`, then remove the setting entirely.
- `mypalace/database.py`: the `after_begin` `SET LOCAL search_path` hook
  becomes **unconditional** (drop the `if settings.tenant_schema_mode ==
  "schema"` guard).
- `mypalace/tenancy.py`: the tenant contextvar must always be populated
  on per-tenant query paths. **Fail closed** — if a per-tenant query
  runs with no tenant in context, raise rather than silently querying
  `public`. (Diagnostics-only behavior from 12.1 is removed.)
- `init_db()`: bootstrap the `default` tenant schema on a fresh DB so the
  zero-config dev experience still boots (design doc §9.3).
- Delete all `table`-mode-only code branches.

### 1.2 Backfill migration (Alembic `0013`)
- 0010 left `tenant_id` NULL in shadow-copied per-tenant rows. Because A
  keeps the `WHERE tenant_id` filters, those filters would under-count
  historical rows.
- New migration: for each tenant in `public.tenants`, for each
  `PER_TENANT_TABLES` entry,
  `UPDATE "<tenant>".<tbl> SET tenant_id = '<tenant>' WHERE tenant_id IS NULL`.
- Idempotent; safe to re-run. Downgrade is a no-op (the column already
  existed).

### 1.3 Cross-tenant aggregation rewrite
After the flip, live data lives in `<tenant>.*`; `public.*` is stale.
These paths currently aggregate over global `public.<table>` and must be
rewritten to **loop tenants, set `search_path` per tenant, aggregate in
Python**:
- `mypalace/api/stats.py` — `tenant_id=ALL` / aggregate (`_all_tenant_ids()`
  already exists; reuse it).
- `mypalace/api/portability.py` — `/v1/admin/export` all-tenants path.
- `mypalace/api/memories.py` — `search?tenant_id=ALL`
  (`memory_service.search_all_tenants`). NB: vector hits already come
  from per-tenant Qdrant collections; the fix is the Postgres metadata
  side.

### 1.4 Deferral (kept intact in v0.12.0)
- `tenant_id` columns on per-tenant tables: **kept.**
- `public.<table>` legacy rows: **kept** (stale after flip; strengthens
  rollback).
- The ~288 `WHERE tenant_id` filters: **kept** (redundant but correct
  once backfilled).
- All of the above move to a future **v0.13.0** cleanup. Documented as
  legacy in the CHANGELOG.

### 1.5 Phase-11 shim removal (bundled)
Delete the deprecated server-side `mypalace-admin` entry point (scheduled
for v0.12.0 in phase 11). The CLI now lives only in
`mypalace-client[cli]`.

### 1.6 Tests
- Make schema mode the **default** integration fixture
  (`tests/integration/conftest.py` `_ensure_schema` hook).
- Cross-tenant-aggregation-under-schema tests for stats / export /
  search `ALL`.
- Backfill-migration (`0013`) test: NULL → populated.
- Fresh-DB `init_db` boot test (default schema bootstrapped).
- Guard test: a per-tenant query with no tenant in context fails closed.

### 1.7 Release
- Version → `0.12.0`; CHANGELOG entry (note the kept-columns deferral and
  the v0.13.0 cleanup); deployment-doc upgrade note + the cutover runbook
  (Part 3).
- Pipeline publishes `mypalace` + `mypalace-client` 0.12.0 to PyPI +
  Docker.

## Part 3 — Cutover runbook (maintenance window, production-grade)

Rollback is cheap **only because writers are stopped and `public.*` is
intact**. Writes during the window would go to `<tenant>.*` only, so a
later downgrade-to-table-mode would lose them — hence stop writers.

0. **Pre-req:** Palace on latest 0.11.x with `0010` applied; export/admin
   tooling reachable.
1. Announce window → **stop mypalclara writers** (or set read-only).
2. **Backup:** `/v1/admin/export` per tenant (NDJSON, stored off-box) +
   record `/v1/admin/stats` row counts (accurate here — still table mode).
3. **Upgrade Palace → 0.12.0:** `alembic upgrade head` runs `0013`
   backfill; schema mode now mandatory.
4. **Verify gate:** post-upgrade `/v1/admin/stats` counts match the
   backup; spot-check `search`; `/health/deep` green.
5. **Pass →** ensure mypalclara is on the 0.12.0 client pin; restart
   writers. Done.
6. **Fail →** downgrade Palace to 0.11.x + set
   `PALACE_TENANT_SCHEMA_MODE=table` (reads intact `public.*`), restart,
   resume mypalclara. Restore-from-backup is the last resort.

*Optional extra caution:* a pre-flip dry-run on 0.11.x with
`flag=schema` + a manual backfill to validate reads before the upgrade.

## Out of scope (this release)

- Dropping `tenant_id` columns / `public.*` rows / the ~288 filters →
  **v0.13.0 cleanup.**
- Online (zero-downtime) cutover.
- Any mypalclara behavior change beyond the client pin.
- Per-tenant databases / per-tenant Postgres roles (design doc §7).

## Follow-up: v0.13.0 cleanup (noted, not planned here)

Once schema mode has soaked in production: Alembic migration to
`DROP COLUMN tenant_id` on per-tenant tables + drop the duplicate
`public.<table>` rows, and delete/rewrite the ~288 redundant filters.
That is the true point of no return (rollback = restore from backup).
